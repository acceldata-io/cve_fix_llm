The scale of the problem — release 3.3.6.4

Manual CVE work does not scale with ODP

One release baseline. Forty-plus components. Thousands of Critical / High / Medium tickets. The same CVE appears again on the next release. This is structural — not a one-off backlog.

And ~70–80% of those tickets are mechanical: already fixed upstream, shaded in someone else’s jar, owned by another platform component, or no fix exists. Engineers should not burn weeks on that work.

The new approach — TeamODP

Judgment stays human. Repetition becomes a system.

01 · GROUND TRUTH

Release-scoped facts

Pull every OSV ticket for one release. Read pinned library versions from GitHub. Never act on the wrong baseline.

02 · DECIDE

Rules first, AI when needed

Deterministic rules close the bulk at ~$0. Hybrid models only for ambiguous triage and hard fixes — not for every ticket.

03 · CONTROL

Human approval gate

Nothing writes to Jira or GitHub without an engineer’s yes. PRs get reviewers. Exceptions get consistent, auditable reasons.

Old way

  • Engineer opens tickets one-by-one
  • Tribal knowledge for “is this fixable?”
  • Inconsistent exception wording
  • Easy to touch the wrong release

TeamODP way

  • Plan the whole release in hours
  • Rules + agent propose; humans approve
  • Uniform justifications + PR links
  • Release + component hard-scoped

Plan
Deterministic FIX vs EXCEPTION for the whole release — zero LLM tokens. Align on common library versions across components.
Verify

Execute
One component at a time. Agent proposes; engineer approves. Build, PR with reviewer, close OSV tickets — release-scoped.
